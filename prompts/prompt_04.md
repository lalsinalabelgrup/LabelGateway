# Normalize requestId vs callId protocol contract

CRITICAL: Respond with TEXT ONLY. Do NOT call any tools.

Define and document the normalised protocol contract between LabelPhone and LabelGateway, separating two concepts that were previously conflated:

## id (request correlation)

- Generated by LabelPhone
- Format: `${command}-${Date.now()}`
- Present in every command envelope
- Used to match command replies to their originating request
- LabelGateway never generates or modifies `id`

## callId (telephony identifier)

- Generated by LabelGateway / the active adapter (never by LabelPhone)
- Identifies a specific call leg
- Present at the **top level** of event envelopes (not nested inside `payload`)
- Present at the **top level** of command envelopes for commands that target an existing call

## Wire formats

Command envelope (client → server):

```json
{ "id": "<uid>", "command": "<name>", "callId": "<cid>", "params": { } }
```

`callId` is present for call-targeting commands (`hangup`, `hold`, `resume`, `answer`, `reject`, `mute`, `unmute`, `setSpeaker`, `transfer`, `sendDTMF`).
`params` is omitted when the command has no parameters.

Command reply (server → client):

```json
{ "id": "<uid>", "result": { } }
{ "id": "<uid>", "error": "<message>" }
```

Push event (server → client):

```json
{ "event": "<name>", "callId": "<cid>", "timestamp": "<ISO8601>", "payload": { } }
```

`callId` is present for all events that belong to a specific call.
`timestamp` is always present in push events.

## Changes required

- Add `timestamp` to all push events (ISO 8601)
- Move `callId` from inside `payload` to top level of event envelopes
- `telephonyGatewayClient.Real._syncState()` must keep `_state.call` in sync in real mode
- Call-targeting commands must pass `_state.call.callId` as the third argument to `_send()`
- `onclose` handler must clear `_state.call`
