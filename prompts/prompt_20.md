==================================================
FEATURE
==================================================

feature/call-hold

The objective is to implement a real telephony Hold / Resume feature.

This is NOT a local audio mute.

This is NOT a transfer.

This is NOT a browser-only feature.

The call must remain established while being placed on hold.

==================================================
IMPORTANT
==================================================

Before writing any code, inspect the current telephony implementation.

Do NOT assume how Hold should be implemented.

Determine how the current provider expects Hold to be negotiated.

Investigate:

- PromoSoft implementation
- SIP signalling
- SDP negotiation
- RTP behaviour
- Existing provider abstraction
- Current audio bridge

Only after understanding the current implementation should code be modified.

==================================================
EXPECTED BEHAVIOUR
==================================================

When the user presses Hold:

- The call remains established.
- The remote party is placed on hold.
- The remote party hears the PBX Music On Hold.
- LabelPhone indicates that the call is on hold.
- No audio is exchanged between both parties while on hold.
- RTP remains stable.
- SIP dialog remains active.

When the user presses Resume:

- Audio immediately resumes.
- Call timers continue.
- UI returns to Active Call.
- No renegotiation failures.
- No call interruption.

==================================================
VERY IMPORTANT
==================================================

The Music On Hold should preferably be generated by the PBX.

Do NOT play an MP3 or browser audio from LabelPhone.

Do NOT fake Hold by muting both directions.

This must be a real telephony Hold.
